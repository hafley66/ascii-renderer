# Session: ascii-renderer-party-walk-params

## Goal
Implement the "party walk" vision: nodes along a path where each node is a distinct scene (landscape, centerpiece, or cluster arrangement) with character and breathing room between them. Expose tuning knobs as CLI args so variants are reproducible.

## Current State
Two commits landed:
- `52a6656` PlantCharacter + NodeMode + party/soup walk modes
- `e35cfcc` PartyParams: expose gap/nodes/scale/detail as CLI args

Working modes:
- `party <theme> [gap] [nodes] [scale] [detail]` -- rejection-sampled non-overlapping boxes, each clipped to a shape mask (ellipse/diamond/rect/trapezoid), color arc per node
- `soup <theme>` -- dense overlapping nodes, no gap enforcement (user liked this as its own thing)

PartyParams: gap (0=auto), nodes (0=auto 5-11), scale (0-100, 50=default), detail (0-100, controls fg elements per landscape node)

## Problem/Context
Magic number churn was the core problem -- every time a constant changed (gap, box count, box size), some seeds looked better and others worse. User directive: stop tweaking constants, expose them as args so every variant is reproducible. Also: never remove existing modes/behaviors, only add new params or modes.

## Solution/Approach
- PlantCharacter struct: branch_factor, taper_dir, taper_strength, landscape_bias, angle_jitter (randomized per walk)
- NodeMode enum: Landscape (3-pass sky+ground+fg), CenterpieceWithSurround, Cluster(arrangement, n)
- ClusterArrangement: Hex, Ring, Loose, Grid
- Node boundary shapes clip all inner layers via mask_intersect
- PartyParams controls the 4 key knobs via CLI args[4..7]

## Key Insights
- User wants ALL variants preserved -- never kill a working mode, only add params/modes
- "soup" was an accident but user loved it, so it stayed as its own mode
- The gap must scale with terminal width or wide monitors get mush
- Exposing numbers as args > finding "the right" constant
- Existing Scene/Layer compositor already handles multi-pass -- nodes just produce richer layer stacks
- Node scenes need visual contrast between layers (sky sparse, ground dense, fg heaviest) to not read as gibberish

- How to handle the "sometimes I want a face explosion in a diamond with sick patterns around it" -- CenterpieceWithSurround exists but needs more dramatic contrast
- Path character between nodes: how thick, what fills, should it be a mode param?
