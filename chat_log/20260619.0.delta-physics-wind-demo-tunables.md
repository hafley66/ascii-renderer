# Session: delta-physics-wind-demo-tunables

## Goal
Make the `delta` ASCII tree animate as a wind-blown articulated tree driven by real physics, expose its tuning knobs through the demo TUI via a declarative config system, and keep playback fast. All work additive; user only runs `demo` mode.

## Current State
All shipped and pushed to main (latest: 3ec9349). 74 tests pass. Working tree clean.
- `draw_delta` is a torsional spring-damper physics sim per branch joint, kinematically chained (trunk motion propagates to twigs), driven by a turbulent wind field. Re-integrates from a bounded warm-up window each frame -> stateless, deterministic, constant-cost regardless of clock. t=0 stays byte-identical (snapshot/static preserved).
- Declarative registry: `ModeSpec { animate: AnimKind, params: &[Param] }`, `AnimKind { Iterate, Vflow, Morph }`, `mode_spec(name)`, `anim_strat(kind)`, `param_f32(key, default)`. delta declares 6 knobs in `DELTA_PARAMS`: stiffness(K), inertia(D), damping(ZETA), wind(WIND), turbulence(TURB), rainbow(RBOW).
- Knobs flow demo -> env `ASCII_P_<KEY>` -> render (subprocess inherits, or in-process reads same env).
- Options pane (`draw_options_pane`) in BOTH the demo browser and the morph player. Toggle `o`; up/down select; left/right (and -/+) adjust; r reset. Auto-opens in the player for modes with knobs.
- Mode picker has a divided-out "✕ cancel" entry pinned at the bottom.
- `iterate_grid(mode, seed, theme, w, h, t)` renders delta IN-PROCESS (no fork). Other iterate modes return None -> fall back to `render_frame_t` subprocess.

## Problem/Context
The morph player's `iterate` strategy forked the whole binary + serialized/parsed a grid every frame (~10ms each at 30fps) -> "runs like piss". delta now in-process fixes it for delta only.

## Solution/Approach
Physics: each Node has parent idx, rest angle rel, len, depth. State arrays theta/omega. Per joint: torque = wind_perp*len*exposure - K*len*theta - c*omega; alpha=torque/I; semi-implicit Euler. I~len^3, k~len => natural freq ~1/len (trunk slow, tips fast). zeta=0.18 underdamped. WARM=6.0, DT=0.08 -> ~75 steps from t0=(t-WARM).max(0). theta clamped ±0.7.
In-process: `iterate_grid` replicates main's palette/rng setup and calls `draw_delta`.

## Key Insights
- Snapshots are CHARS ONLY (grid_to_string uses .ch) -> color knobs like RBOW never affect snapshots. Char-affecting changes gated by `windy = t != 0.0` keep t=0 byte-identical.
- Stateless frames + bounded warm-up window = constant-cost physics that looks like persistent simulation (transient from artificial rest start decays within WARM, leaving true forced steady-state).
- Rust 2024: std::env::set_var/remove_var are unsafe; wrapped in unsafe (demo loop single-threaded). Child processes inherit parent env, so set_var in morph_session feeds both subprocess and in-process renders.
- Subprocess capture pattern (ASCII_GRID_DUMP + ASCII_GRID_W/H + ASCII_T env) is how morph gets a Grid for any mode without refactoring dispatch; in-process iterate_grid is the perf escape hatch.
- mode_spec is the single source of truth for animate strategy (replaced hardcoded match) and knob declarations.
