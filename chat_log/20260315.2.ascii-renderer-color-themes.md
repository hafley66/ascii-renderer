# Session: ascii-renderer-color-themes

## Goal
Continue building the ascii-renderer: add new pattern primitives (stepped frets, Aztec diamond, masonry), add color support via crossterm, and create a named theme system.

## What Was Built This Session
1. **Stepped fret primitive** (xicalcoliuhqui): turtle-walk spiral, arm length sequence n,n,n-1,n-1,...,1,1. Box-drawing corners. Border band tiling along edges. Works but could use connecting logic between adjacent units and interlocking pairs.
2. **Aztec diamond primitive**: domino shuffling algorithm (move/destroy/fill). Rewrote from HashMap doubled-coords to dense 2D array. Diamond shape renders correctly, arctic circle visible but glyph readability is poor in monochrome -- needs color to distinguish quadrants. User said "looks off" -- shelved visual polish until color.
3. **Color system**: Cell{ch, fg, bg} replacing char. crossterm for ANSI escape generation. Every primitive takes color params. Trees lighten at deeper branches. Flowers brighten at center.
4. **Theme system**: 11 hand-tuned named palettes across warm/cool/mono/vivid categories. Swatch mode for comparison. Seed-based HSL rotation as default.
5. **RESEARCH.md**: 12-section reference covering all primitives, algorithms in pseudocode, glyph harmony rules, comparative tables by role/complexity/cost, all source links.

## Key Insights
- Stepped fret arm sequence is n,n,n-1,n-1,...,1,1 (each length twice, two turns per rectangle side)
- Aztec diamond collision detection: after MOVE, any adjacent S-above-N or E-left-of-W pair is a collision (legitimate pairs are always 3 apart after move)
- Half-block glyphs (▀▄▌▐) need color to be legible -- monochrome Aztec diamond reads as noise
- Terminal color = ANSI escape sequences injected in output stream, ~20 bytes overhead per color change, run-length optimization important
- crossterm is thin wrapper over escape code formatting, used for Color type and SetForegroundColor/SetBackgroundColor
